# T1 — Написание тест-кейсов

# 1. Цель тестирования

Проверить корректность работы REST API метода получения статуса продукта.

API должно:

- корректно обрабатывать обязательные и необязательные параметры
- корректно выполнять авторизацию
- корректно обрабатывать комбинации параметров
- возвращать корректные HTTP статусы
- возвращать корректный JSON
- корректно обрабатывать ошибки

# 2. Объект тестирования

Структура запроса:

GET `/products/{productId}/status?authToken=&recalculate=&owner=&region=`

Параметры запроса:

| Параметр    | Тип     | Обязательность | Возможные значения                    |
| ----------- | ------- | -------------- | ------------------------------------- |
| authToken   | string  | обязательный   | строка из 16 символов (буквы и цифры) |
| recalculate | boolean | необязательный | true / false / null                   |
| owner       | string  | необязательный | Создатель / Пользователь / null       |
| region      | string  | необязательный | Северо-Запад / Сибирь / Поволжье      |

Особенности:

- параметры recalculate, owner и region необязательные
- параметры не коррелируют между собой

# 3. Ожидаемые ответы API

Успешный ответ:

HTTP 200

JSON
{
  "productStatus": 1
}

Допустимые значения:

- 1 : продукт готов
- 0 : продукт не готов

Ошибки:

- 401 : отсутствует или неверный authToken
- 404 : продукт не найден
- 500 : внутренняя ошибка сервера

# 4. Стратегия тестирования

Тестирование включает следующие направления:

4.1 Функциональное тестирование

Проверка корректности работы endpoint.

4.2 Проверка параметров

Проверяются:

- обязательный параметр authToken
- необязательные параметры
- комбинации параметров

4.3 Pairwise testing

Используется pairwise-подход для проверки комбинаций параметров.

Это позволяет покрыть большинство дефектов при меньшем количестве тестов.

4.4 Граничные значения

Для authToken проверяются:

- длина < 16
- длина = 16
- длина > 16
- только цифры
- только буквы
- спецсимволы

4.5 Негативное тестирование

Проверяются:

- неверные значения параметров
- отсутствующие параметры
- неправильный формат
- неожиданные значения

4.6 Проверка формата ответа

Проверяется:

- корректность JSON
- типы данных
- наличие обязательных полей

# 9. Основные тест-кейсы

## TC-01 Получение статуса продукта

Запрос:

GET /products/100/status?authToken=ABCD1234EFGH5678

Ожидаемый результат:

HTTP 200

Ответ:

{
  "productStatus": 1
}

## TC-02 Проверка всех параметров

Запрос:

GET /products/100/status?authToken=ABCD1234EFGH5678&recalculate=true&owner=Создатель&region=Сибирь

Ожидаемый результат:

HTTP 200

Ответ:

{
  "productStatus": 1
}

## TC-03 Продукт не найден

Запрос:

GET /products/999999/status?authToken=ABCD1234EFGH5678

Ожидаемый результат:

HTTP 404

## TC-04 Отсутствует authToken

Запрос:

GET /products/100/status

Ожидаемый результат:

HTTP 401

## TC-05 Неверный authToken

Запрос:

GET /products/100/status?authToken=123

Ожидаемый результат:

HTTP 401

## TC-06 Проверка ошибки сервера

Шаги:

- Искусственно вызвать внутреннюю ошибку сервера.
- Отправить запрос GET /products/100/status?authToken=ABCD1234EFGH5678

Ожидаемый результат:

HTTP 500

Ответ:

{
  "errorMessage": "Internal server error"
}

# 10. Дополнительные проверки

Проверить:

- передачу null значений
- чувствительность к регистру
- корректность структуры JSON
- нагрузочную устойчивость API
