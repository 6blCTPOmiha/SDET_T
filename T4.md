# T4 — Составление плана тестирования

# 1. Цель тестирования

Цель тестирования — проверить корректность обработки запросов на управление услугой (подключение, отключение, блокирование), корректность взаимодействия между сервисами A и B через брокер сообщений RabbitMQ, а также соблюдение порядка выполнения команд.

# 2. Объект тестирования

В системе присутствуют:

- Сервис A
  - принимает REST-запросы:
    - подключение услуги
    - отключение услуги
    - блокирование услуги
  - отправляет команды сервису B через AMQP
  - обеспечивает упорядочивание команд
  - принимает REST-запросы независимо от статуса выполнения предыдущих операций
  - изменяет статус услуги (только после положительного ответа от сервиса B)

- Сервис B
  - принимает AMQP сообщения
  - выполняет операции с услугой
  - отправляет результат выполнения операции обратно через AMQP

- RabbitMQ
  - брокер сообщений между сервисами

# 2. Области тестирования

В рамках тестирования проверяются следующие области системы:

- REST API сервиса A
- отправка AMQP сообщений сервисом A
- получение AMQP сообщений сервисом B
- отправка ответов сервисом B
- обработка ответов сервисом A
- механизм упорядочивания команд
- изменение статуса услуги
- обработка ошибок взаимодействия
- обработка параллельных REST запросов

# 4. Области, не подвергаемые тестированию

В рамках данного тестирования не проверяются:

- внутренняя бизнес-логика сервисов A и B
- производительность системы под высокой нагрузкой
- безопасность REST API
- масштабируемость RabbitMQ
- пользовательские интерфейсы (UI), если таковые имеются

# 5. Риски

Возможные риски при работе системы:

- потеря сообщений в брокере RabbitMQ
- нарушение порядка обработки команд
- отсутствие ответа от сервиса B
- некорректная обработка ошибок
- рассинхронизация статусов услуги
- зависание обработки команд при отсутствии ответа

# 6. Расписание

Примерное расписание тестирования:

| Этап                          | Описание                          | Срок   |
| ----------------------------- | --------------------------------- | ------ |
| Подготовка среды              | Развертывание сервисов и RabbitMQ | 1 день |
| Подготовка тестовых сценариев | Определение тест-кейсов           | 1 день |
| Проведение тестирования       | Выполнение тестов                 | 2 дня  |
| Анализ результатов            | Анализ дефектов и отчёт           | 1 день |

# 7. Критерии начала тестирования

Тестирование может быть начато при выполнении следующих условий:

- сервис A развернут и доступен
- сервис B развернут и принимает сообщения
- RabbitMQ работает и доступен
- REST API сервиса A доступен
- тестовая среда подготовлена

# 8. Критерии приостановки тестирования

Тестирование может быть приостановлено при возникновении следующих условий:

- недоступность сервисов
- недоступность RabbitMQ
- критические ошибки, блокирующие выполнение тестов
- отсутствие тестовой среды

# 9. Критерии возобновления тестирования

Тестирование может быть возобновлено после:

- устранения критических дефектов
- восстановления доступности сервисов
- восстановления работоспособности RabbitMQ

# 10. Критерии окончания тестирования

Тестирование считается завершенным при выполнении следующих условий:

- выполнены все запланированные тест-кейсы
- критические дефекты устранены
- система корректно выполняет основные операции
- подготовлен отчёт о результатах тестирования

# 11. Результаты тестирования

По итогам тестирования должны быть подготовлены следующие результаты:

- отчёт о выполнении тестирования
- список обнаруженных дефектов
- список выполненных тест-кейсов
- рекомендации по улучшению системы (при необходимости)

# 12. Стратегия тестирования

Тестирование проводится на уровне интеграционного тестирования и включает:

- проверку REST API сервиса A
- проверку взаимодействия сервисов через RabbitMQ
- проверку корректности последовательности команд
- проверку обработки положительных и отрицательных ответов
- проверку поведения системы при ошибках

Основной подход:

- функциональное тестирование
- негативное тестирование
- тестирование граничных сценариев
- тестирование параллельных запросов

# 13. Верхнеуровневые тест-кейсы

| ID    | Название                                           | Описание                                                         | Ожидаемый результат                                                                         |
| ----- | -------------------------------------------------- | ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| TC-01 | Подключение услуги                                 | Проверка обработки запроса на подключение услуги                 | Сервис A отправляет команду сервису B, после положительного ответа статус услуги изменяется |
| TC-02 | Отключение услуги                                  | Проверка обработки запроса на отключение услуги                  | Команда отправляется сервису B, после ответа статус услуги изменяется                       |
| TC-03 | Блокирование услуги                                | Проверка обработки запроса на блокирование услуги                | Сервис A отправляет команду сервису B и обновляет статус после ответа                       |
| TC-04 | Порядок команд                                     | Проверка, что сервис A отправляет команды строго последовательно | Следующая команда отправляется только после получения ответа                                |
| TC-05 | Параллельные REST запросы                          | Проверка обработки нескольких REST запросов одновременно         | Все запросы принимаются, команды выполняются последовательно                                |
| TC-06 | Отрицательный ответ сервиса B                      | Проверка обработки ошибки выполнения операции                    | Статус услуги не изменяется                                                                 |
| TC-07 | Отсутствие ответа сервиса B                        | Проверка поведения системы при отсутствии ответа                 | Следующая команда не отправляется                                                           |
| TC-08 | Доставка сообщений                                 | Проверка передачи сообщений через RabbitMQ                       | Сообщения успешно доставляются между сервисами                                              |
| TC-09 | Обработка ответа                                   | Проверка корректной обработки ответа сервиса B                   | Статус услуги обновляется корректно                                                         |
| TC-10 | Очередность команд при большом количестве запросов | Проверка корректной обработки длинной очереди команд             | Команды выполняются в порядке поступления                                                   |
