# T3 — Составление тест-плана

# 1. Цель тестирования

Цель тестирования — проверить корректность работы REST API, которое рассчитывает стоимость покупки продукта на основе количества приобретаемых товаров, а также корректность обработки авторизации и ошибок.

API должно:

- корректно обрабатывать обязательные и необязательные параметры
- корректно выполнять авторизацию
- корректно обрабатывать комбинации параметров
- возвращать корректные HTTP статусы
- возвращать корректный JSON
- корректно обрабатывать ошибки

# 2. Объект тестирования

Структура запроса:

GET `/products/{productId}/cost?authToken=&count=&start=`

Параметры запроса:

| Параметр  | Описание                                      |
| --------- | --------------------------------------------- |
| productId | идентификатор продукта                        |
| authToken | токен авторизации (16 символов буквы + цифры) |
| count     | количество приобретаемых товаров              |
| start     | индекс начала вычисления                      |

# 3. Ожидаемые ответы API

| Код | Ответ                                 |
| --- | ------------------------------------- |
| 200 | { "cost": number }                    |
| 401 | при неверном или отсутствующем токене |
| 404 | если продукт не найден                |
| 500 | { "errorMessage": "..." }             |

# 4. Области подвергаемые тестированию

Тестируются следующие области системы:

- Авторизация (authToken)
- Проверка входных параметров
- Расчёт стоимости
- Граничные значения параметров
- Логика партий по 5 товаров
- Обработка ошибок
- Тело ответа JSON
- HTTP статус коды

# 5. Области не подвергаемые тестированию

В рамках текущего тестирования не проверяются:

- UI (отсутствует)
- Внутренняя реализация сервиса
- База данных
- Производительность системы
- Безопасность инфраструктуры

# 6. Риски

| Риск                            | Описание                                       |
| ------------------------------- | ---------------------------------------------- |
| Недоступность тестового сервера | невозможно выполнить тестирование              |
| Отсутствие тестовых данных      | невозможно проверить разные статусы продуктов  |
| Изменение API                   | тест-кейсы могут устареть                      |
| Недостаточная документация      | возможна неправильная интерпретация требований |
| Неверная логика расчёта         | ошибка в формуле стоимости                     |

# 8. Критерии начала тестирования

Тестирование может быть начато если:

- API endpoint развернут
- доступна тестовая среда
- есть доступ к системе авторизации
- подготовлены тестовые данные

# 9. Критерии приостановки тестирования

Тестирование приостанавливается если:

- API недоступно
- критические дефекты блокируют тестирование
- отсутствуют необходимые тестовые данные

# 10. Критерии возобновления тестирования

Тестирование возобновляется если:

- восстановлена работа API
- исправлены блокирующие дефекты
- предоставлены необходимые данные

# 11. Критерии окончания тестирования

Тестирование считается завершенным если:

- выполнены все тест-кейсы
- исправлены критические дефекты
- подготовлен отчёт о тестировании

# 12. Результаты тестирования

Результатом тестирования являются:

- отчёт о выполнении тест-кейсов
- список найденных дефектов
- отчёт о покрытии требований
- итоговый тестовый отчёт

# 13. Стратегия тестирования

Тестирование включает следующие направления:

13.1 Функциональное тестирование

Проверка корректности работы endpoint.

13.2 Проверка параметров

Проверяются:

- параметры: {productId}, authToken, count, start
- результирующая общая стоимость

13.3 Граничные значения

Для authToken проверяются:

- длина < 16
- длина = 16
- длина > 16
- только цифры
- только буквы
- спецсимволы

13.4 Негативное тестирование

Проверяются:

- неверные значения параметров
- отсутствующие параметры
- неправильный формат
- неожиданные значения

13.5 Проверка формата ответа

Проверяется:

- корректность JSON
- типы данных
- наличие полей

# 14. Инструменты тестирования
